────────────────────────────────────────────
AUTHENTICATION SYSTEM
────────────────────────────────────────────

POST /auth/login
- Body: { email, password }
- Validates credentials against users collection
- Returns: { token, user: { uid, name, email, profile, organization_id } }
- Token payload: { uid, email, profile, organization_id, iat, exp }
- Token expiry: 7 days

Middleware (auth.js):
- Reads token from header: x-access-token
- Verifies with JWT_SECRET from .env
- Attaches decoded user to req.user

Middleware (roleGuard.js):
- Usage: roleGuard(['superAdmin', 'organization', 'lead manager'])
- Reads req.user.profile
- Returns 403 if role not in allowed list

────────────────────────────────────────────
API ROUTES — HEADERS (Dynamic Table Columns)
────────────────────────────────────────────

GET    /headers/:organization_id/:module
  → Returns header config for that org + module
  → If none exists, returns default column config for that module

POST   /headers
  → Body: { organization_id, module, columns[] }
  → Creates header config (one per org per module)

PUT    /headers/:organization_id/:module
  → Body: { columns[] }
  → Replaces column config (full update)

PATCH  /headers/:organization_id/:module/column/:key
  → Body: { visible, order, width, label, pinned }
  → Updates a single column property

DELETE /headers/:organization_id/:module
  → Resets to system default for that module

────────────────────────────────────────────
API ROUTES — FORM FIELDS (Dynamic Forms)
────────────────────────────────────────────

GET    /formFields/:organization_id/:form_name
  → Returns form field config for that org + form
  → If none exists, returns system default field config

POST   /formFields
  → Body: { organization_id, form_name, fields[] }
  → Creates form config

PUT    /formFields/:organization_id/:form_name
  → Body: { fields[] }
  → Full replacement of form config

PATCH  /formFields/:organization_id/:form_name/field/:key
  → Body: { label, required, visible, order, options, grid_cols, ... }
  → Updates a single field's config

DELETE /formFields/:organization_id/:form_name
  → Resets form to system default

────────────────────────────────────────────
LEAD TRANSFER API (replaces Firebase transferLead function)
────────────────────────────────────────────

POST /leads/transfer
Auth: JWT required
Role: lead manager, team lead, organization

Body:
{
  leadsData: [{ ...leadObject, contactId: "mongoId" }],
  owner: { email, uid, organization_id },
  reason: "string",
  leadType: "Leads" | "Data",
  options: {
    fresh: true/false,
    notes: true/false,
    attachments: true/false,
    task: true/false,
    contactDetails: true/false
  }
}

Logic:
1. Validate role from req.user.profile
2. For each lead:
   a. Create NEW contacts document for new owner
   b. Update old document: transfer_status=true, transfer_reason, associate_status
   c. Set audit trail: previous_owner_1/2, transfer_by_1/2, previous_stage_1/2
   d. If options.fresh=true → reset stage to FRESH, clear follow-up fields
   e. If options.task=true → copy Pending tasks to new lead, cancel old ones
   f. If options.notes or options.attachments → copy contactResources
3. Send notifications (email + in-app via Socket.IO)
4. Return { success: true, newLeadIds: [] }

SOCKET.IO — REAL-TIME EVENTS
────────────────────────────────────────────

Events to emit:
  lead:assigned   → { leadId, toUid, fromUid }
  lead:updated    → { leadId, changes }
  lead:transferred → { oldLeadId, newLeadId, toUid }
  task:created    → { taskId, leadId, uid }
  notification:new → { message, uid, type }

Room strategy:
  Each user joins room: organization_id + uid
  Lead managers join: organization_id room
  Emit to specific user: io.to(uid).emit(...)
  Emit to org:           io.to(organization_id).emit(...)

DYNAMIC FORM — DynamicForm.tsx
────────────────────────────────────────────

1. On mount: GET /formFields/:org_id/:form_name

3. FieldRenderer maps field_type:
   text       → <input type="text">
   number     → <input type="number">

   textarea   → <textarea>
   checkbox   → <Checkbox>
   radio      → <Radio.Group> with options
   file       → <Upload>
4. Layout: Bootstrap 12-col grid using field.grid_cols
5. Validation: apply field.validation rules to React Hook Form register
6. Props: { formName, onSubmit, defaultValues, readOnly }

ROLE-BASED UI RULES
────────────────────────────────────────────

superAdmin:
  - Can see all orgs, all users, all leads
  - Access to system settings

organization:
  - Sees only own org data
  - Can manage users, headers, form fields

lead manager:
  - Sees leads of own team (self + team leads + sales under them)
  - Can transfer leads to team lead or sales
  - Cannot access org settings

team lead:
  - Sees leads of own sales executives + self
  - Can transfer only to own sales executives

sales:
  - Sees only own assigned leads
  - Cannot transfer leads
  - Cannot access settings

Hide/show sidebar menu items and action buttons based on req.user.profile.

REPORTING_TO — DYNAMIC DROPDOWN
────────────────────────────────────────────

On User Create/Edit form:
1. Read the selected role for the new user
2. Based on role, fetch valid managers:
   - sales        → GET /users?profile=team lead&org=:id
   - team lead    → GET /users?profile=lead manager&org=:id
   - lead manager → GET /users?profile=organization&org=:id
   - organization → GET /users?profile=superAdmin
3. Populate reporting_to dropdown with { label: name, value: uid }
4. This is the dynamic dropdown for manager assignment.
